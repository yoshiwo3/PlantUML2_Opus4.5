## 改訂戦略の要点

### 対象読者

「スキルアップを目指している非エンジニア」
- 専門用語を「排除」されたくない（学びたい）
- 「なぜ必要か」の文脈を求めている
- 最終的には実務で使えるレベルを目指している

### 核心原則

1. **削らない、足す**: 技術詳細は維持し、理解の補助を追加
2. **段階的開示**: 概要→詳細→技術仕様の3段階（`<details>`タグ活用）
3. **文脈優先**: 「何か」の前に「なぜ必要か」を説明
4. **アナロジー活用**: 技術概念を日常の例えで補足
5. **視覚化強化**: Mermaid図、アイコン（📌🎬✅⚠️📘）、色分け

### ドキュメント構造

```
Part 0: ナビゲーション（新設）
  ├── 読み方ガイド
  ├── システム全体像
  └── 用語集

Part 1: 機能一覧（改訂）
  └── 各機能を3層構造で記述
      ├── 📌 この機能について（概要）
      ├── 🎬 操作の流れ（手順）
      └── 📘 技術詳細（折りたたみ）

Part 2: 全体整合性（図解強化）
Part 3: 技術リファレンス（現状維持）
```

   - 対応: シンプルなテキストラベルに変更 ✅

   - 問題: Obsidian等で表示されない
