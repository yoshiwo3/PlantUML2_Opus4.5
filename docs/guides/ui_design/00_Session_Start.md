## 0.5 検証フェーズについて（重要）

> **⚠️ 現在のステータス**: 憲法v3.3は **Draft（検証中）** です。

### 検証フェーズとは

UI設計図表開発憲法は理論的設計に基づいて作成されましたが、**実際のワイヤーフレーム作成で検証されていません**。最初の1-3画面の作成を「検証」として実施し、発見した問題を即座に憲法に反映します。

### 検証モードでの作業

| 項目 | 通常モード | 検証モード（現在）|
|------|-----------|------------------|
| 目的 | 成果物作成 | **プロセス検証 + 成果物作成** |
| 問題発見時 | 知見として記録 | **即座に憲法を更新** |
| 統計更新 | 月次集計 | **都度更新** |
| 完了条件 | 90点以上 | 90点以上 + **検証レポート作成** |

### 検証レポート項目

各画面の作成完了時、以下を記録：

```markdown
## 検証レポート: [画面名]

### プロセス検証結果
- [ ] 協調作成フロー（Step 1-3）は機能したか
- [ ] 評価基準は適切だったか
- [ ] 5パスレビューは有効だったか

### 発見した問題
| # | 問題 | 対応 | 憲法更新 |
|:-:|------|------|:--------:|
| 1 | ... | ... | §X.X |

### 改善提案
- ...
```

### 検証完了条件

以下を満たした時点で憲法を v1.0-stable に昇格：

- [ ] 3画面以上のワイヤーフレームを作成完了
- [ ] 協調作成プロセスの有効性を確認
- [ ] 統計データが蓄積された（問題発生率が計測可能）
- [ ] 致命的なプロセス欠陥がない

## 1. 作業開始前の確認（必須）

### 1.1 次の画面を特定する

**`active_context.md` の「UI設計図表 詳細進捗（19画面）」テーブルを読み、以下の優先順位で次の画面を特定する：**

1. **ワイヤーフレーム🔴 かつ MVP優先度** → 最優先
2. **ワイヤーフレーム🔴 かつ Phase 2優先度** → 次点
3. **全て✅** → 作業完了

**特定ロジック**:
```
active_context.md の「UI設計図表 詳細進捗」セクションを参照

FOR each row in 認証画面群 → メイン画面群 → モーダル群 → 管理画面群:
    IF ワイヤーフレーム == 🔴 AND 優先度 == MVP:
        RETURN この画面を次の作業対象として提案
    END IF
END FOR

IF MVP全完了:
    FOR each row:
        IF ワイヤーフレーム == 🔴 AND 優先度 == Phase 2:
            RETURN この画面を次の作業対象として提案
        END IF
    END FOR
END IF

RETURN "全画面完了"
```

### 1.2 自動開始禁止

**作業を自動開始してはならない。**

以下の情報をユーザーに提示し、**承認を得てから**作業を開始すること。

### 1.3 ユーザーへの確認内容

```

**対象画面**: #N XX画面（カテゴリ: auth/main/modal/admin）
**関連UC**: UC X-X（〇〇）

  - UC図: §X

**TD-015準拠確認**:
  - 低精度（Low-Fidelity）: [確認済み]
```

### 1.4 承認後のみ作業開始

## 3. 作業手順

### Step 1: 参照ドキュメント確認

`01_Reference_Guide.md` §2 で対象画面の参照先を確認し、以下を読む：

- UC図（該当セクション）
- 業務フロー図（該当セクション）
- 機能一覧表（該当機能ID）
- 既存ワイヤーフレーム（スタイル参照）

### Step 2: 画面一覧確認（憲法 §1.3.6）

- 画面一覧テンプレートで対象画面を確認
- 対応UC・業務フロー整合性テーブルを確認
- 3点対比（画面一覧↔ワイヤーフレーム↔遷移図ノード）を意識

### Step 3: Evidence作成

```powershell
# 現在時刻を確認
Get-Date -Format "yyyyMMdd_HHmm"

# Evidenceディレクトリ作成
pwsh scripts/create_evidence.ps1 <yyyyMMdd_HHmm>_ui_design_<screen_name>
```

### Step 4: Todoリスト作成

`TodoWrite`ツールで以下を登録：

```
1. 関連ドキュメント確認（UC図、業務フロー図、機能一覧表）
2. TD-015原則確認
3. ワイヤーフレーム作成（Excalidraw）
4. 5パスレビュー（AI自己評価）
5. PNGエクスポート → ユーザーに提示
6. ユーザーレビュー待ち
7. [ユーザー編集後] AI厳格評価・採点
8. [90点達成まで] イテレーションループ
9. 画面遷移図更新（必要時）
10. active_context.md更新
11. SERENA Memory保存
12. Git commit
```

### Step 4.5: ワイヤーフレーム一覧確認

既存ワイヤーフレームがある場合、`00_wireframe_index.md` で一覧確認する。

```markdown
# index.md の形式（EX-005 Embedded Index Pattern）

## 01. ログイン画面
![[01_login/default.excalidraw]]

## 02. ダッシュボード
![[02_dashboard/default.excalidraw]]
![[02_dashboard/search_active.excalidraw]]
```

- **閲覧**: `00_wireframe_index.md`（スクロールで全画面確認）
- **編集**: 個別の `.excalidraw` ファイル

> **詳細**: UI_Design_Knowledge_Base.md §EX-005

### Step 5: AI初期作成

`02_Authoring_Guide.md` のチェックリストに従う：

- [ ] TD-015原則遵守（低精度、手書き風、グレースケール）
- [ ] 命名規則遵守（NN_category_screen_name.excalidraw）
- [ ] UI要素配置（UC/業務フロー図に基づく）
- [ ] アクセシビリティ考慮（憲法 §8）

### Step 6: ユーザーレビュー・手直し

1. AIがPNGをユーザーに提示
2. ユーザーの選択を待つ：
   - **「OK」** → Step 7へ（90点以上確認後）
   - **「修正指示」** → AIが修正 → Step 5へ戻る
   - **「自分で編集」** → ユーザーがObsidian Excalidrawで編集 → 完了通知後Step 7へ

### Step 7: AI厳格評価（ユーザー編集後）

| Pass | 対象 | 確認内容 |
|:----:|------|---------|
| 1 | TD-015準拠 | 低精度、手書き風、グレースケール |
| 2 | 網羅性 | 画面一覧チェックリスト照合 |
| 3 | UI要素 | UC図・業務フロー図と照合 |
| 4 | 過剰詳細 | 「やりすぎ」チェック（憲法 §3.4） |
| 5 | 整合性 | 他画面との一貫性 |

**評価・採点結果**:
- **90点以上** → 「合格です。次のステップに進みます」→ Step 8へ
- **90点未満** → 問題点を詳細報告 → Step 6へ戻る（イテレーションループ）

> **反復回数**: 無制限（90点達成まで継続）

### Step 8: 成果物保存

1. SVGエクスポート（正式版）
2. 正式版SVG保存（`docs/proposals/diagrams/10_wireframe/`）
3. **⚠️ `00_wireframe_index.md` 更新（必須）** ← 進捗サマリー、作成計画、完了済み画面セクション
4. 画面遷移図更新（該当時）
5. active_context.md更新（ワイヤーフレーム欄を✅に）
6. 00_raw_notes.md更新（操作ログ追記）
7. work_sheet.md更新（成果物記録）
8. SERENA Memory保存
9. Git commit

> **⚠️ 更新漏れ防止**: Step 8の1-7は**すべて必須**。特に`00_wireframe_index.md`は一覧管理の要なので漏れなく更新すること。
